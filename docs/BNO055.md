# BNO055 Module Documentation

## Overview

The `BNO055` module integrates the DFRobot BNO055 9-axis absolute orientation sensor. The class handles sensor initialization, runtime configuration, calibration persistence, and exposes convenient helpers for retrieving fused orientation data, raw acceleration/gyro/magnetometer values, linear acceleration, gravity vectors, and temperature readings.

## Configuration

Configure the module in [config/bno055.yml](config/bno055.yml). The file supports multiple instances using the shared `instances` list. Each entry is passed as keyword arguments to the module constructor.

```yaml
bno055:
  enabled: true               # Enable the module loader to instantiate BNO055 modules
  path: modules.bno055.BNO055  # Import path for the class
  instances:
    - name: body_imu
      bus: 1                   # I2C bus number
      address: 0x28            # Sensor I2C address (default 0x28, optional 0x29)
      operation_mode: NDOF     # Operating mode (see Operation Modes below)
      power_mode: NORMAL       # Power mode
      debug_on_boot: false     # When true, publishes all readings on system/loop/1 ticks
      external_crystal: true   # Enable when a 32.768 kHz crystal is present (default on DFRobot board)
      calibration_file: config/bno055_body_calibration.json
      units:
        acceleration: mps2
        angular_rate: dps
        euler: degrees
        temperature: c
```

### Dependencies

Install required packages before enabling the module:

- System package: `python3-smbus`
- Python package: `smbus`

Run `./install.sh` to process dependency installation automatically.

### Multiple Instances

Add additional entries under `instances` for each IMU. Use unique `name` values to avoid collisions in the module loader. The loader will expose each instance via the pattern `BNO055_<instance_name>`.

## Operation Modes

The module accepts any of the BNO055 operation modes documented in the datasheet. Common options:

- `NDOF`: Full sensor fusion (default).
- `IMU`: Accelerometer + gyroscope fusion without magnetometer.
- `COMPASS`: Magnetometer + accelerometer fusion for heading.
- `ACCONLY`, `GYROONLY`, `MAGONLY`: Raw sensor modes.

Switch modes at runtime using `set_operation_mode("MODE")` if required.

## Usage

Once enabled, interact with the IMU through the messaging bus. Each instance listens on `bno055/<instance_name>/...` topics and publishes responses on predefined `bno055/<instance_name>/data/*` and `bno055/<instance_name>/event/*` channels.

```python
# Request fused Euler angles
messaging_service.publish('bno055/body_imu/read/euler')

# Subscribe to the default response topic
messaging_service.subscribe('bno055/body_imu/data/euler', handle_orientation)

# Retrieve quaternion data
messaging_service.publish('bno055/body_imu/read/quaternion')
# -> response on bno055/body_imu/data/quaternion

# Change operation mode
messaging_service.publish(
  'bno055/body_imu/set/operation_mode',
  mode='IMU'
)
# -> event on bno055/body_imu/event/operation_mode

# Persist calibration to file configured in the instance
messaging_service.publish('bno055/body_imu/calibration/save')
# -> event on bno055/body_imu/event/calibration_save

# Load calibration from an explicit list of 22 bytes
messaging_service.publish(
  'bno055/body_imu/calibration/load',
  calibration=calibration_bytes
)
# -> event on bno055/body_imu/event/calibration_load
```

Common response topics include:

- `bno055/<name>/data/euler`
- `bno055/<name>/data/quaternion`
- `bno055/<name>/data/acceleration`
- `bno055/<name>/data/gyroscope`
- `bno055/<name>/data/magnetometer`
- `bno055/<name>/data/linear`
- `bno055/<name>/data/gravity`
- `bno055/<name>/data/temperature`
- `bno055/<name>/data/status`
- `bno055/<name>/data/calibration_status`
- `bno055/<name>/data/calibration`
- `bno055/<name>/event/operation_mode`
- `bno055/<name>/event/power_mode`
- `bno055/<name>/event/calibration_save`
- `bno055/<name>/event/calibration_load`

### Messaging Service Logging

The module relays log messages via the project messaging bus once `ModuleLoader.set_messaging_service` wires the dependency. Logging calls are skipped gracefully when the bus is not yet available (for example, during bootstrap).

## Calibration Workflow

The BNO055 stores calibration profiles internally while powered. Persist calibration data to disk to avoid repeating the procedure after every reboot.

1. Enable the module with `calibration_file` pointing to a writable JSON path (example: `config/bno055_body_calibration.json`).
2. Move the sensor through the manufacturer-recommended calibration routine until `imu.is_fully_calibrated()` returns `True`.
3. Persist calibration bytes:

   ```python
   imu.save_calibration_to_file('config/bno055_body_calibration.json')
   ```

4. On subsequent boots the module automatically loads the file and applies offsets via `load_calibration_from_file`.

Call `imu.read_calibration_data()` to inspect the raw 22-byte payload if needed.

## Error Handling

I2C errors bubble up as `OSError` exceptions from the underlying SMBus library. Surround high-frequency polling loops with try/except blocks when running outside the module’s helper methods. Use `get_system_status()` to inspect sensor self-test and `get_calibration_status()` to verify individual subsystem readiness.

## Additional Notes

- `external_crystal: true` enables the sensor’s onboard oscillator. Disable only if the crystal is physically absent.
- `debug_on_boot: true` subscribes the module to `system/loop/1`, automatically re-publishing every available reading on each loop tick for quick diagnostics.
- Adjust `units` mapping to match downstream processing expectations. Supported values are `mps2` or `mg`, `dps` or `rad/s`, `degrees` or `radians`, and `c` or `f` for temperature.
- Use `set_power_mode("LOW_POWER")` or `set_power_mode("SUSPEND")` to reduce consumption when the IMU is idle.
